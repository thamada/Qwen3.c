# `qwen3-8b/xdna2/xdna-gemv/kernels/` ディレクトリとは何か（初心者向け）

このディレクトリ（**`qwen3-8b/xdna2/xdna-gemv/kernels/`**）には、`qwen3-xdna2` が **AMD Ryzen AI の NPU（XDNA2）** に行列計算を任せるときに読み込む **制御用バイナリ**を置く想定でしたが、**このリポジトリが直接配布しているのは「本物の計算カーネル」ではありません**。まず全体像から説明します。

**親パッケージ `qwen3-8b/xdna2/xdna-gemv/`** については、[上位の README](../README.md) を参照してください（ツールチェイン手引き・スタブ生成スクリプトへの案内）。

---

## 0. この README で一番伝えたいこと（ctrlcode だけ先に）

**ctrlcode（制御コード）** は、ざっくり言うと **「NPU の中で動く“オーケストレーション用のプログラム”」のバイト列**です。

| 普通のプログラム | ctrlcode（この文脈） |
|------------------|----------------------|
| CPU が実行する機械語やバイトコード | **NPU ファームウェア上の ERT** が解釈して実行する **低レベル命令の連なり** |
| `main()` を書いてコンパイル | AMD の **MLIR-AIE / IRON / Peano** などのツールで **チップ向けに生成** |
| ソースが読めることが多い | **生のバイナリ**で、人間が読むより **ハードウェアと DMA のタイミングを書いたもの**に近い |

**このプロジェクトでは**、その ctrlcode を **`bf16-gemv-<n>x<d>.bin` という1ファイルにそのまま保存**し、実行時に NPU 側の **Instruction Buffer（命令用メモリ領域）** に載せてから **`ERT_START_NPU`** で走らせます。

ここまでピンとこなくて大丈夫です。**次のセクションで「なぜそれが必要か」をホスト CPU との役割分担つきで説明します。**

---

## 1. そもそも LLM の推論で何をしているか

大規模言語モデル（このプロジェクトでは Qwen3 系）は、ざっくり言うと **大量の行列とベクトルの掛け算・足し算**を繰り返します。

- **線形層（行列ベクトル積）**  
  長さ `n` のベクトル `x` と、形状 `[出力次元 d × 入力次元 n]` の重み行列 `W` があるとき、  
  `y = W x`（各出力要素が `W` の行と `x` の内積）を **GEMV（General Matrix-Vector multiply）** と呼ぶことがあります。
- **ファイル名の `bf16-gemv-<n>x<d>.bin`** の **`n` と `d`** は、この **`x` の長さ `n`** と **出力ベクトル `y` の長さ `d`** に対応します（実装がその形状の GEMV 用カーネルを選ぶため）。

つまり、このフォルダにある名前は「**このサイズの重み・活性に対応する NPU 用プログラム**」という意味のラベルです。

## 2. NPU を使うとき、ホストと NPU は何を分担しているか（ctrlcode の位置づけ）

### 2.1 身近なイメージで置き換える

- **ホスト（Linux 上の `qwen3-xdna2`）**  
  GGUF を読む、Attention や RMSNorm など **この実装では CPU で回している部分**もある、**GEMV のたびに入力バッファを用意する**、ドライバへ「実行して」と頼む——といった **司令塔**。
- **NPU（XDNA2 の AIE アレイ）**  
  **行列×ベクトルなど、用意されたデータに対して SIMD 的に計算する工場**。
- **ctrlcode**  
  「どのメモリから DMA でデータを引いて、どのタイルに載せ、どの順で計算し、結果をどこに書き戻すか」を **NPU 側の実行環境が理解できる形式で書いたレシピ（バイナリ）**。

GPU で例えるなら、**カーネル本体に相当するのはオーバーレイ／コンフィグ側**で、**ctrlcode は「そのカーネルを正しい順序・正しいメモリ転送で動かすディスパッチ用スクリプト」**に近いイメージです（厳密には別物ですが、**「計算そのもの」と「制御・転送」が分かれている**という感覚は近いです）。

**GPU（ROCm/HIP）との本格的な対比と、「ユーザーがカーネルをプログラムできるか」は §3** にまとめています。

### 2.2 このリポジトリのコードが実際にしていること（名前がつながると楽）

Linux の **`amdxdna` ドライバ**（`/dev/accel/...`）経由で次のような流れになります（ファイル先頭コメントの要約です）。

1. **ハードウェアコンテキスト**を開く（空間パーティションとして列を確保するイメージ）。
2. **バッファオブジェクト（BO）**を作る——入力ベクトル・出力・重みの scratch などが **ホストと NPU が共有できるメモリ**として載る。
3. **ctrlcode のバイト列**を **Instruction Buffer（DEV_HEAP、最大 64 MiB 級の「ctrlcode 用アリーナ」）** に書き込む。
4. **`EXEC_CMD`** で **`ERT_START_NPU`** を送ると、ファームウェア上の **ERT** がその ctrlcode（**XAie_TxnOpcode の列**として記述されたバイナリ）を実行する。
5. ctrlcode の実行の過程で **ホスト DDR と AIE 列のメモリの間の DMA** が動き、**計算タイル上で BF16 GEMV** が進む。

だから **ctrlcode は「C で書いた関数を gcc でコンパイルした .o」ではなく**、**チップとファームウェアが約束したフォーマットの、転送＋同期＋演算トリガのシーケンス**です。

### 2.3 なぜ `(n,d)` ごとにファイルが分かれるのか

GEMV の **行列の大きさが変わると**、

- **DMA で運ぶブロックの切り方**
- **タイルに載せるデータの並び**
- **ループやパイプラインの組み方**

が変わります。その結果、**「この形状専用にツールチェインが生成した ctrlcode」**がファイル単位で用意される、という理解で十分です。**同じ Qwen3 でも層によって `(n,d)` が違う**ので、`bf16-gemv-4096x4096.bin` と `bf16-gemv-4096x151936.bin` のように **名前が複数ある**わけです。

### 2.4 公式ドキュメントへのリンク

概要は AMD の説明資料（例: [amdnpu.rst](https://github.com/amd/xdna-driver/blob/main/src/driver/doc/amdnpu.rst)）にも **ctrlcode とホスト DDR 間の DMA** の話がまとまっています。

## 3. GPU カーネルとの対比で見る XDNA の「カーネル」

この節では **ROCm / HIP での GPU カーネル** を基準に、XDNA（Ryzen AI NPU）側の **「カーネル」が何を指すか** と、**ユーザーが GPU と同じ感覚でプログラムできるか** を整理します。

### 3.1 GPU（ROCm / HIP）側でいう「カーネル」

AMD GPU で典型的な流れは次のイメージです。

- **ソースを書く**  
  HIP（C++ にカーネル記法が載ったもの）などで **`__global__` カーネル関数**を書く。
- **コンパイルする**  
  `hipcc` などで **ISA に近い GPU 向けバイナリ**へコンパイルする。
- **ホストから起動する**  
  **グリッド／ブロック／共有メモリ**などを指定して **`kernel<<<...>>>(args)`**（または同等 API）で **何度でも**起動できる。
- **ユーザーの自由度**  
  ループ構造、レジスタ・メモリの使い方、同期の入れ方など、**ハードの制約の範囲でアルゴリズムをかなり自由に記述できる**のが普通です。

つまり ROCm / HIP の文脈では **「カーネル」＝ユーザーがソースから生成し、ホストが都度ディスパッチする計算プログラム**という意味が強いです。

### 3.2 XDNA 側でこの README が呼んでいる「カーネル」

XDNA2 の実行モデルでは、GPU の「1本の HIP カーネル」に相当するものが **一枚岩ではなく分割**されています（`qwen3-8b/xdna2/main.c` 冒頭コメントの整理に沿います）。

| かたまり | 役割のイメージ |
|----------|----------------|
| **オーバーレイ（CU 設定など）** | AIE の **どのリソースをどう使うか**の構成情報。コンテキスト作成後にドライバ経由で **CONFIG_HWCTX** などに載せるイメージ。 |
| **ctrlcode（`.bin`）** | **DMA の順序・同期・演算のキック**などを **トランザクション列**で書いたもの。Instruction Buffer に載り **`ERT_START_NPU`** で実行される。 |
| **ホスト側ランタイム** | BO の用意、`EXEC_CMD`、同期オブジェクト——GPU でいう **カーネル起動とバッファ管理**に相当。 |

ここでいう **「XDNA のカーネル」** は、口頭では **オーバーレイ＋ctrlcode（＋それを動かすホスト手続き）のセット** を指すことが多く、**HIP の `__global__` 関数1つと 1:1 対応しません**。

### 3.3 ユーザーは GPU（ROCm/HIP）のようにプログラムできるか（必読）

**結論を先に書きます。**

- **GPU の HIP カーネルのように、「普段の意味でのユーザー主導のプログラミング」とはかなり違う** のが典型です。  
  多くの開発者は **HIP のソースを書いてその場でコンパイルして載せる**のではなく、**AMD が提供するツールチェイン（MLIR-AIE / IRON / Peano など）から生成された成果物**を **ファイルとして配置・ロード**する立場になります。
- **一方で「一切プログラムできない」わけではありません。**  
  **ツールチェインと LLVM / MLIR スタックに踏み込めるユーザー／ベンダ**は、高レベル IR や DSL から **自分の演算をコンパイルして ctrlcode＋オーバーレイを生成する**道があります。ただしそれは **HIP で書くような対話的・短文法的なカーネル開発**とは別レイヤで、**データフロー／タイル／DMA を意識したコンパイラ作業**に近いです。

整理すると次のようになります。

| 観点 | GPU（ROCm / HIP） | XDNA（このプロジェクトが想定する使い方） |
|------|-------------------|------------------------------------------|
| **何を書くか** | HIP のカーネル関数（C++ 風） | 直接は **XAie トランザクション列のバイナリ**ではなく、多くは **ツールが生成した `.bin`（ctrlcode）とオーバーレイ** |
| **誰がコンパイルするか** | 開発者が `hipcc` 等でビルド | **コンパイラ／リンカがツールチェイン側**（開発者は設定・IR・パイプラインに関与することが多い） |
| **ランタイムでの差し替え** | ビルド済みカーネルを **任意のパラメータで何度でも launch** | **形状 `(n,d)` ごとなど**に **事前ビルドされた ctrlcode ファイル**をマッチさせる（このリポジトリは `bf16-gemv-<n>x<d>.bin`） |
| **「ユーザーがプログラム可能か」** | **標準的ワークフローで明確に Yes**（カーネルソースがユーザー資産） | **HIP と同じ意味での Yes ではない**。実効的には **ツールチェイン利用者／ベンダ向けに Yes（間接的）**、**推論アプリ統合者には「バイナリ消費」が主** |

**このリポジトリでやっていること（初心者向けに言い換えると）:**  

- **`qwen3-xdna2` は「NPU 向けの計算レシピを作るソフト」ではなく、「レシピファイルがあれば NPU に載せて GEMV を頼む推論プログラム」**です。そのため **本物の ctrlcode（レシピ）は同梱せず**、代わりに **名前だけ合わせたスタブ**が **このディレクトリ（`kernels/`）** にあります（再生成は [スタブ生成スクリプト](../gen-xdna-gemv-stubs.py)）。  
- **実機で速くしたい人は**、別途ツールチェインでビルドした **本物の `.bin`** を、環境変数 **`XDNA_GEMV_DIR`** が指すフォルダに置く想定です（置かないままだと、上の表の **「バイナリ消費」が主**の読者でも、そのバイナリが無いので **CPU で同じ計算をする**ことになります）。  
- **スタブだけでは NPU が速くならない理由**は単純で、**中身が「計算の手順書」ではなくプレースホルダだから**です。プログラムはスタブを検出すると **NPU に送らず**、間違った命令で壊さないよう **CPU に任せます**。GPU でいえば **実行可能なカーネルバイナリを置かず、ダミーファイルだけがある状態**に近く、**チップはあるがその経路は使われない**イメージです。

## 4. 用語ミニ辞典（ctrlcode まわりだけ）

| 用語 | ひとこと |
|------|----------|
| **ERT** | NPU ファームウェア側で、ユーザーが渡したコマンド（ctrlcode の実行など）を処理する実行時コンポーネント。 |
| **Instruction Buffer / insn heap** | ctrlcode を載せるための **NPU コンテキスト付きのメモリ領域**。ここに `.bin` の中身がコピーされる。 |
| **XAie_TxnOpcode** | AIE を「設定する・DMA を動かす・同期する」などの **トランザクション命令**。ctrlcode はこれらの **列**として表現されることが多い。 |
| **`ERT_START_NPU`** | 「この ctrlcode を実行せよ」と ERT に伝えるための **実行コマンドの種類**のひとつ。 |
| **`bf16-gemv-<n>x<d>.bin`** | 上記の **ctrlcode をファイルにしたもの**（本物の場合）。このリポジトリのデフォルトは **別物のスタブ**（後述）。 |

## 5. このリポジトリに入っている `.bin` は「スタブ（プレースホルダ）」です

本ディレクトリに同梱されている `bf16-gemv-*.bin` は、すべて **たった 64 バイトの小さなファイル**です。

- **目的**  
  - `qwen3-xdna2` が探す **ファイル名・ディレクトリ構成**を、ビルドや説明用にリポジトリ内で固定する。  
  - `./xdna2/qwen3-xdna2 ... --xdna-status`（`qwen3-8b` から）で **`[STUB]`** と表示され、「ファイルはあるが、実機向けカーネルではない」ことが一目で分かるようにする。
- **先頭のマジック文字列 `GQF3XDNA`**  
  - プログラムがこれを検出すると **NPU にそのバイナリを送らず**、あたかもファイルが無かったときと同様に **CPU 側の計算にフォールバック**します（誤って無効な命令を NPU に流さないため）。

**初心者の方へのメッセージ:**  
ここにあるファイルだけを置いた状態では、**「NPU が勝手に速くなる」わけではありません**。  
速くするには、後述のとおり **ツールチェインでビルドした本物の ctrlcode に差し替える**必要があります。

## 6. 環境変数 `XDNA_GEMV_DIR` と `--xdna-status` の見方

- **`XDNA_GEMV_DIR`**  
  `bf16-gemv-<n>x<d>.bin` を探す **親ディレクトリ**です。  
  例: リポジトリをクローンしたうえで `qwen3-8b` でビルド済みバイナリを動かす場合、よく使うのは次のような指定です。

  ```bash
  export XDNA_GEMV_DIR="$(pwd)/xdna2/xdna-gemv/kernels"
  ```

  （カレントがリポジトリ直下なら `$(pwd)/qwen3-8b/xdna2/xdna-gemv/kernels` で構いません。）

- **`--xdna-status`（または `-X`）**  
  モデルを軽く開いて **各形状のファイルがあるか・スタブか**を一覧します。
  - **`[STUB]`** … この README が説明している **リポジトリ同梱プレースホルダ**（または同じ形式のファイル）。**中身は ctrlcode として実行されません**。
  - **`[ OK ]`** … スタブではない **読み取り可能なバイナリ**。**ファイルが読めるだけ**であり、**必ずしも正しい ctrlcode とは限りません**が、少なくとも「スタブ拒否」は通過します。
  - **`MISS`** … その名前のファイルが無いか読めない。

また **ドライバやハードウェアコンテキストが開けない環境**では、そもそも NPU 経路に入れないため、`--xdna-status` の説明文では「ハードウェア GEMV は動かない」と表示されることがあります。それでも **`[STUB]` / `MISS`** の一覧は、ファイル準備の確認に使えます。

## 7. Qwen3-VL-8B（テキスト経路・既定形状）とファイル名の対応

このモデル設定では、線形層の GEMV で次の **5 種類の `(n,d)`** が使われます（`wq` と `wo` が同じファイル名を共有します）。

| ファイル名 | おおよその役割（レイヤーまわり） |
|-----------|----------------------------------|
| `bf16-gemv-4096x4096.bin` | 注意機構の Q 投影・出力投影など |
| `bf16-gemv-4096x1024.bin` | K/V 投影など |
| `bf16-gemv-4096x12288.bin` | フィードフォワードの gate / up |
| `bf16-gemv-12288x4096.bin` | フィードフォワードの down |
| `bf16-gemv-4096x151936.bin` | 語彙サイズへの logits（語彙に広がる巨大な一次変換） |

**別のモデルや別の `dim` / `hidden` を読み込んだ場合**は、この表とは異なる `(n,d)` が `--xdna-status` に出ることがあります。そのときは **表示された形状に合わせて** `.bin` を追加・命名し直す必要があります。

## 8. 本物のカーネルに差し替えるには（ここから先が本番）

スタブを **実際の NPU 計算に使える ctrlcode** に置き換えるには、次が必要です。

1. **Linux 側で `amdxdna` が動き、`CREATE_HWCTX` まで成功する**（ファームウェアや権限、`render` グループなど）。  
2. **各 `(n,d)` について、ツールチェインが生成したバイナリ**を、同じファイル名で `XDNA_GEMV_DIR` に配置する。

具体的なビルド手順はチップ世代・ドライバ・ツールの組み合わせで変わるため、**コマンド列付きの詳細**は **[`toolchain/README.md`](../toolchain/README.md)**（Xilinx mlir-aie / AMD IRON / Peano / `aiecc` に沿ったセットアップと Qwen 形状と `GEMV(M,K)` の対応、`--aie-generate-npu-insts` 等）に記しています。概要としては MLIR-AIE / xdna-driver / IREE 周りの公式資料への参照に加え、同文書を読んでください。

### 8.1 既にホストされているバイナリがある場合（このディレクトリの `Makefile`）

チームや第三者が **同じファイル名**で本物の `.bin` を HTTPS で公開している場合、このディレクトリの **`Makefile`** でまとめて取得できます。

```bash
cd qwen3-8b/xdna2/xdna-gemv/kernels
make
```

別のミラーにする場合のみベース URL を指定します。

```bash
make XDNA_GEMV_BIN_URL_BASE='https://別の公開先/'
```

Makefile 先頭の **`XDNA_GEMV_BIN_URL_BASE ?= ...`** に既定の GitHub Releases 形式の URL を書いてあり、`make` のみでも取得を試みます。**まだ Releases に同名アセットが無いときは `curl` が 404 で失敗**します。そのときは URL を自分のサーバなどに編集して再実行するか、上記のように変数で上書きしてください。

ベース URL は **末尾 `/` あってもなくても可**（Makefile が `/` を補います）。その場所に次の **5 ファイル**がそのまま置いてある必要があります（[スタブ生成スクリプト](../gen-xdna-gemv-stubs.py) が出力する名前と同名）。

- `bf16-gemv-4096x4096.bin`
- `bf16-gemv-4096x1024.bin`
- `bf16-gemv-4096x12288.bin`
- `bf16-gemv-12288x4096.bin`
- `bf16-gemv-4096x151936.bin`

用法は **`make help`** も参照。

`curl` または `wget` のどちらかが必要です。取得後は `export XDNA_GEMV_DIR="$(pwd)"`（またはクローンルートからの相対パス）で `qwen3-xdna2` を実行します。

### 8.2 MLIR-AIE / IRON で自前ビルドする場合

**Ubuntu 上で XRT と mlir-aie の wheel を揃え、`aiecc --aie-generate-npu-insts` で NPU 用トランザクションバイナリを出す**までの手順・注意点・参照 URL の一覧は **[`toolchain/README.md`](../toolchain/README.md)** にまとめています（`qwen3-xdna2` の **ioctl 単体ランタイム**と公式の **XRT 検証パス**の違い、`bf16-gemv-*.bin` との完全一致が保証されない境界も同文書で説明しています）。

差し替えが成功すると、`--xdna-status` で **`[ OK ]`** が並び、実際の推論ログでは **NPU GEMV の呼び出し回数**が増える期待になります（ハードウェアとファームが正常な場合）。

## 9. スタブファイルの再生成（開発・検証用）

リポジトリルートで:

```bash
python3 qwen3-8b/xdna2/xdna-gemv/gen-xdna-gemv-stubs.py
# 別ディレクトリへ出力する場合:
python3 qwen3-8b/xdna2/xdna-gemv/gen-xdna-gemv-stubs.py /path/to/dir
```

または `qwen3-8b` で:

```bash
make gen-xdna-kernels
```

いずれも出力されるのは **あくまでスタブ**であり、上記のとおり **実機向け ctrlcode にはなりません**。

---

## 10. 実行例（状態確認だけ）

```bash
cd qwen3-8b
export XDNA_GEMV_DIR="$(pwd)/xdna2/xdna-gemv/kernels"
./xdna2/qwen3-xdna2 /path/to/model.gguf --xdna-status
```

すべて **`[STUB]`** であれば、「名前と場所は合っているが、**NPU が実行する ctrlcode はまだ置かれていない**」状態です。それがこのディレクトリのデフォルトの意味です。

## 11. 処理の流れ（俯瞰ダイアグラム）

```text
  ┌─────────────────────────────────────────────────────────────┐
  │ ホスト: qwen3-xdna2                                          │
  │ ・GEMV ごとに入力 BO を用意し、重みを BF16 scratch に載せる    │
  │ ・bf16-gemv-<n>x<d>.bin（ctrlcode）を Instruction Buffer へ  │
  │ ・EXEC_CMD / ERT_START_NPU で「実行」とドライバへ依頼           │
  └────────────────────────────┬────────────────────────────────┘
                               │ ioctl（amdxdna）
                               ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ NPU ファームウェア: ERT が ctrlcode（XAie_TxnOpcode 列）を実行 │
  │ ・DDR ⇄ AIE 列メモリの DMA                                     │
  │ ・AIE タイル上で BF16 GEMV                                     │
  └─────────────────────────────────────────────────────────────┘
```

この「右下の箱」の **左から右へのレシピ** が **ctrlcode** と捉えると、`README` とソースコメントの語が一本につながります。
