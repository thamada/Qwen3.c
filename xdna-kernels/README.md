# `xdna-kernels/` ディレクトリとは何か（初心者向け）

このフォルダには、`qwen3-xdna2` が **AMD Ryzen AI の NPU（XDNA2）** に行列計算を任せるときに読み込む **制御用バイナリ**を置く想定でしたが、**このリポジトリが直接配布しているのは「本物の計算カーネル」ではありません**。まず全体像から説明します。

---

## 1. そもそも LLM の推論で何をしているか

大規模言語モデル（このプロジェクトでは Qwen3 系）は、ざっくり言うと **大量の行列とベクトルの掛け算・足し算**を繰り返します。

- **線形層（行列ベクトル積）**  
  長さ `n` のベクトル `x` と、形状 `[出力次元 d × 入力次元 n]` の重み行列 `W` があるとき、  
  `y = W x`（各出力要素が `W` の行と `x` の内積）を **GEMV（General Matrix-Vector multiply）** と呼ぶことがあります。
- **ファイル名の `bf16-gemv-<n>x<d>.bin`** の **`n` と `d`** は、この **`x` の長さ `n`** と **出力ベクトル `y` の長さ `d`** に対応します（実装がその形状の GEMV 用カーネルを選ぶため）。

つまり、このフォルダにある名前は「**このサイズの重み・活性に対応する NPU 用プログラム**」という意味のラベルです。

---

## 2. NPU を使うとき、プログラムは何をロードするのか

`qwen3-xdna2` は Linux の **`amdxdna` ドライバ**（`/dev/accel/...`）経由で NPU とやり取りします。

NPU 側には **ファームウェア上の実行環境（ERT など）**があり、ユーザが用意するのはざっくり次のようなものです。

- **入力・出力・重み**が置かれたメモリ（バッファオブジェクト）へのポインタ  
- **そのバッファの間でデータを動かし、演算を進めるための「極めて低レベルな命令列」**  
  → ドキュメントでは **ctrlcode（制御コード）** と呼ばれることが多く、このプロジェクトでは **`bf16-gemv-<n>x<d>.bin` にその内容をそのまま保存**している想定です。

重要な点です。

- それは **Python や通常の C で書いた関数をコンパイルしただけのファイルではありません**。  
- **AMD のツールチェイン（MLIR-AIE / IRON / Peano など）で、対象チップ向けに生成されたバイナリ**であり、中身は **チップ専用のトランザクション／オペコード列**です。
- そのため **「適当なバイト列を置けば動く」ものではなく**、形状 `(n,d)` ごとに **正しくビルドされたもの**が必要です。

概要は AMD の説明資料（例: [amdnpu.rst](https://github.com/amd/xdna-driver/blob/main/src/driver/doc/amdnpu.rst)）にも ctrlcode とホスト DDR 間の DMA の話がまとまっています。

---

## 3. このリポジトリに入っている `.bin` は「スタブ（プレースホルダ）」です

本ディレクトリに同梱されている `bf16-gemv-*.bin` は、すべて **たった 64 バイトの小さなファイル**です。

- **目的**  
  - `qwen3-xdna2` が探す **ファイル名・ディレクトリ構成**を、ビルドや説明用にリポジトリ内で固定する。  
  - `./qwen3-xdna2 ... --xdna-status` で **`[STUB]`** と表示され、「ファイルはあるが、実機向けカーネルではない」ことが一目で分かるようにする。
- **先頭のマジック文字列 `GQF3XDNA`**  
  - プログラムがこれを検出すると **NPU にそのバイナリを送らず**、あたかもファイルが無かったときと同様に **CPU 側の計算にフォールバック**します（誤って無効な命令を NPU に流さないため）。

**初心者の方へのメッセージ:**  
ここにあるファイルだけを置いた状態では、**「NPU が勝手に速くなる」わけではありません**。  
速くするには、後述のとおり **本物の ctrlcode に差し替える**必要があります。

---

## 4. 環境変数 `XDNA_GEMV_DIR` と `--xdna-status` の見方

- **`XDNA_GEMV_DIR`**  
  `bf16-gemv-<n>x<d>.bin` を探す **親ディレクトリ**です。  
  例: リポジトリをクローンしたうえで `qwen3-8b` でビルド済みバイナリを動かす場合、よく使うのは次のような指定です。

  ```bash
  export XDNA_GEMV_DIR="$(pwd)/../xdna-kernels"
  ```

  （カレントがリポジトリ直下なら `$(pwd)/xdna-kernels` で構いません。）

- **`--xdna-status`（または `-X`）**  
  モデルを軽く開いて **各形状のファイルがあるか・スタブか**を一覧します。
  - **`[STUB]`** … この README が説明している **リポジトリ同梱プレースホルダ**（または同じ形式のファイル）。
  - **`[ OK ]`** … スタブではない **読み取り可能なバイナリ**。実機で本当に動くかは、中身が正しい ctrlcode かどうかに依存します。
  - **`MISS`** … その名前のファイルが無いか読めない。

また **ドライバやハードウェアコンテキストが開けない環境**では、そもそも NPU 経路に入れないため、`--xdna-status` の説明文では「ハードウェア GEMV は動かない」と表示されることがあります。それでも **`[STUB]` / `MISS`** の一覧は、ファイル準備の確認に使えます。

---

## 5. Qwen3-VL-8B（テキスト経路・既定形状）とファイル名の対応

このモデル設定では、線形層の GEMV で次の **5 種類の `(n,d)`** が使われます（`wq` と `wo` が同じファイル名を共有します）。

| ファイル名 | おおよその役割（レイヤーまわり） |
|-----------|----------------------------------|
| `bf16-gemv-4096x4096.bin` | 注意機構の Q 投影・出力投影など |
| `bf16-gemv-4096x1024.bin` | K/V 投影など |
| `bf16-gemv-4096x12288.bin` | フィードフォワードの gate / up |
| `bf16-gemv-12288x4096.bin` | フィードフォワードの down |
| `bf16-gemv-4096x151936.bin` | 語彙サイズへの logits（語彙に広がる巨大な一次変換） |

**別のモデルや別の `dim` / `hidden` を読み込んだ場合**は、この表とは異なる `(n,d)` が `--xdna-status` に出ることがあります。そのときは **表示された形状に合わせて** `.bin` を追加・命名し直す必要があります。

---

## 6. 本物のカーネルに差し替えるには（ここから先が本番）

スタブを **実際の NPU 計算に使える ctrlcode** に置き換えるには、次が必要です。

1. **Linux 側で `amdxdna` が動き、`CREATE_HWCTX` まで成功する**（ファームウェアや権限、`render` グループなど）。  
2. **各 `(n,d)` について、ツールチェインが生成したバイナリ**を、同じファイル名で `XDNA_GEMV_DIR` に配置する。

具体的なビルド手順はチップ世代・ドライバ・ツールの組み合わせで変わるため、この README では **コマンド一式まで固定できません**。公式・コミュニティの MLIR-AIE / xdna-driver / IREE 周りの資料を参照してください。

差し替えが成功すると、`--xdna-status` で **`[ OK ]`** が並び、実際の推論ログでは **NPU GEMV の呼び出し回数**が増える期待になります（ハードウェアとファームが正常な場合）。

---

## 7. スタブファイルの再生成（開発・検証用）

リポジトリルートで:

```bash
python3 tools/gen-xdna-gemv-stubs.py
# 別ディレクトリへ出力する場合:
python3 tools/gen-xdna-gemv-stubs.py /path/to/dir
```

または `qwen3-8b` で:

```bash
make gen-xdna-kernels
```

いずれも出力されるのは **あくまでスタブ**であり、上記のとおり **実機向けカーネルにはなりません**。

---

## 8. 実行例（状態確認だけ）

```bash
cd qwen3-8b
export XDNA_GEMV_DIR="$(pwd)/../xdna-kernels"
./qwen3-xdna2 /path/to/model.gguf --xdna-status
```

すべて **`[STUB]`** であれば、「名前と場所は合っているが、まだ本物の ctrlcode ではない」状態です。それがこのディレクトリのデフォルトの意味です。
